# Fleets Scale Performance Report

最終更新: 2026-03-08

## 1. サマリ

- 256機体まで動作確認済み（100/128/200/256で実行実績）。
- CPUの主負荷は `sim` と `vsp`（ほぼ頭打ち）、機体数増加で顕著に伸びるのは `bridge`。
- ホストCPUはピーク時も概ね余力あり（`host cpu_p95` は最大でも約 `80%` 前後）。
- 多プロセス化（例: 2proc→8proc）は、同一条件で wall time 改善を確認（約8%短縮）。

## 4. 実行時間比較（show-runner, 200機）

ユーザー実測:

- 1proc: `real 232.65`, `user 39.38`, `sys 23.40`
- 2proc: `real 171.87`, `user 39.94`, `sys 22.18`
- 4proc: `real 135.62`, `user 41.59`, `sys 17.13`
- 6proc: `real 146.22`, `user 51.37`, `sys 13.96`
- 8proc: `real 158.10`, `user 56.73`, `sys 17.58`
- 12proc: `real 159.23`, `user 62.38`, `sys 18.33`
- 16proc: `real 172.77`, `user 62.26`, `sys 20.53`

解釈:

- wall time は短縮（1proc `232.65` → 4proc `135.62`, 約42%改善）。
- 1proc → 2proc でも大幅改善（約26%）。
- 4proc が最速。
- `user` 増は並列度を使えていることと整合。
- 6proc以降は悪化傾向で、8proc以降は改善が鈍化/悪化。
- 16proc は `real` が悪化し、過並列によるオーバーヘッド増加が示唆される。
- 改善幅は限定的で、初期化/登録系の待ちが残る。

暫定結論:

- 本条件での最適な `drone-service` プロセス数は **8〜12proc**。
- 本条件での最速は **4proc**（現時点の実測範囲）。

## 5. ボトルネック所見

- `sim` はほぼ 2コア相当で頭打ち（`cpu_p95 ≈ 200%`）。
- `vsp` はほぼ 1コア相当で頭打ち（`cpu_p95 ≈ 100%`）。
- `bridge` は機体数増加とともに増加（100機: `50.505%` → 256機: `81.340%`）。
- ホスト全体は余力があるが、局所的な待ち・初期化同期の影響で体感遅延が出る。

## 6. 安定運用メモ

- `asset_timeout_usec` は `hakoniwa-core-pro` 側で反映される修正が必要（`hako_pro.cpp`）。
- 大規模時は `set_ready/takeoff` の投入方式（batch/init concurrency/retry）が成功率に影響。
- 200機以上は起動タイミングのゆらぎが出やすく、再試行前提の運用が必要。

## 7. 次アクション

1. `set_ready` の段階的投入戦略を標準化（推奨バッチ/同時度を固定）。
2. 2/4/6/8/12/16 proc で同一シナリオを再計測し、wall time と成功率を並記。
3. bridge のCPU増加要因（送受信/変換/配信）を関数別にプロファイルして最適化候補を抽出。
4. 16proc を同条件で追加測定し、8/12/16proc の改善飽和点を確定する。
